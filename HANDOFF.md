# Junebug CLI handoff

## Purpose and current state

This repository is an early Rust implementation of **Junebug CLI**, a local-first agentic coding CLI. The full product roadmap is in [PLAN.md](PLAN.md). Do not represent the project as feature-complete: the core agent loop is working, but major v0.2ŌĆōv1 capabilities remain unimplemented.

The project lives at <https://github.com/weeksdev/junebug-cli> (main branch; tags `v*` trigger the release workflow). The product, binary, crate, and repository were renamed from Febo to Junebug in July 2026. New state is written under `.junebug`/`~/.junebug`, with read compatibility for legacy `.febo` data. Avoid broad rewrites until you understand the safety boundaries below.

## What works today

- Real streamed REST calls through OpenAI-compatible Chat Completions endpoints:
  - `openrouter` (default), via `OPENROUTER_API_KEY`
  - `openai`, via `OPENAI_API_KEY`
  - `deepseek`, via `DEEPSEEK_API_KEY`
- Direct Anthropic Messages API support via `ANTHROPIC_API_KEY`, including SSE text/tool streaming, model discovery, canonical `tool_use`/`tool_result` transcript translation, usage, cancellation, and provider switching mid-session.
- Local Ollama support via provider `ollama` (alias `local`): Junebug auto-detects the runtime at `OLLAMA_HOST` or `http://127.0.0.1:11434`, requires no API key, lists installed models through `/v1/models`, and reuses the OpenAI-compatible streaming/tool-call adapter at `/v1/chat/completions`. Ollama requests set `reasoning_effort: none` so Qwen3 emits responsive tool calls instead of a long hidden thinking trace. Ollama models participate in the unified `/model` picker, direct `ollama:model` selection, sticky sessions, routing, and all three swarm roles. The default/fallback model name is the tool-capable `qwen3:8b`; Junebug never pulls a model automatically.
- Generic local OpenAI-compatible support via `local-openai` (aliases `openai-local`, `lmstudio`, `vllm`): it activates only when `LOCAL_OPENAI_BASE_URL` is set and its `/v1/models` endpoint is reachable, with optional `LOCAL_OPENAI_API_KEY`. Like Ollama it participates in model selection, routing, sessions, and swarms; when no model is pinned it chooses the first returned local model.
- OpenAI-compatible request histories are repaired at the provider edge by dropping legacy empty assistant entries (`content: null` with no tool calls), new empty streamed turns are rejected before entering history, and DeepSeek `reasoning_content` is preserved for thinking-mode tool continuation but stripped when switching to providers that may reject it.
- Cloud API keys can be supplied through environment variables, a local ignored `.env` file, or the user-level `~/.junebug/credentials.env` written by `junebug set --provider NAME API_KEY` (0600 on unix; the command saves the key and then starts the REPL on that provider). Lookup order: process env, workspace `.env`, user credentials file. Supported names are `openrouter`, `openai`, `deepseek`, `anthropic` (`claude` alias), and `ollama` (`local` alias; no key).
- Model-driven multi-turn tool loop with streamed text, tool calls, tool results, token usage, JSONL headless output, and local JSONL sessions. `MAX_TURNS` is 1000 ŌĆö a runaway-loop backstop only, never a task limiter (it was 12 early on; that throttled real tasks and was deliberately raised).
- An interactive REPL (`junebug` with no prompt and a terminal attached) with a Claude Code-style UI:
  - agent turns run on a worker thread (`std::thread::scope`) while the main thread renders `TurnEvent`s from an mpsc channel: streamed Markdown (`src/markdown.rs`, line-buffered ANSI styling), a braille spinner with elapsed seconds, `ŌÅ║ tool(args)` / `ŌÄ┐ result` activity lines, and approval prompts (raw mode is dropped for the y/N read, then restored);
  - **Esc (or Ctrl-C) interrupts a running turn**: a shared `AtomicBool` is checked per SSE line in the provider (partial tool calls are discarded, a `[response interrupted by user]` marker is appended to the recorded assistant text) and before each tool execution (pending calls get `ERROR: interrupted by user` results so tool_call pairing stays valid);
  - a raw-mode line editor (`src/editor.rs`) with completion menus: `/` completes slash commands, `@` searches workspace files (name-prefix ranked, hidden/target/node_modules skipped, 2000-file/8-depth caps); Tab/Enter accept, Ōåæ/Ōåō navigate menu or input history; Ctrl-A/E/U/K editing; falls back to plain `read_line` without a terminal;
  - `@path` mentions attach the file contents (via the policy-guarded `Workspace::read_file`) to the outgoing user message in `<attached-file>` tags;
  - slash commands: `/help`, `/status`, `/diff`, `/changes` (read-only full-screen changed-file list and per-file diff; native Git when present, otherwise latest shadow checkpoint), `/explorer` (searchable/collapsible workspace tree and syntax-highlighted file viewer; `e` suspends the TUI and opens the selected file in `$VISUAL`/`$EDITOR` ŌĆö multi-word values like `code --wait` work, fallback `vi`/`notepad` ŌĆö with a best-effort checkpoint taken first so `/rewind` covers manual edits; protected paths and symlinks never appear in the tree via `should_hide`, and the result/error shows as a yellow footer notice); both browsers use `ŌåÆ` to focus the selected file/diff, `Ōåæ`/`Ōåō` to scroll it, and `ŌåÉ` to return to the tree. The `/changes` diff pane stays read-only. Pane fitting is display-column-aware (`browser::char_columns`, a pragmatic dependency-free range table): emoji/CJK count as two columns and combining marks/variation selectors as zero, so lines containing them no longer overflow the pane and wrap rows into the tree column. Also `/exit`, `/compact` (summary replaces in-memory history; session log keeps raw history), `/model` (no-arg opens one arrow-key menu grouped by every available providerŌĆöcredentialed cloud APIs plus reachable OllamaŌĆöand lists each provider's live model results below it; selecting a model switches provider in one step; `/model provider:model` remains available), and `/permissions` (arrow-key `select_menu` over read-only/ask/workspace-write/yolo, switchable mid-session; rebuilds the `PolicyEngine` each turn from a mutable `permission` local in `repl`);
  - a colored status badge under the prompt always shows the effective access level; Shift+Tab cycles `read-only ŌåÆ ask ŌåÆ workspace-write ŌåÆ yolo` in place without losing typed input and also works during a running turn, whose spinner carries the same colored mode. A shared atomic `PermissionState` feeds the live `PolicyEngine`; each tool snapshots it at dispatch so changes apply to the next tool rather than racing halfway through one. Plan mode remains visibly locked read-only;
  - `editor::select_menu(title, choices, initial)` is the shared raw-mode arrow-key picker (up/down or j/k, enter/tab accept, esc/ctrl-c cancel), used by `/model` and `/permissions`; it renders a terminal-height scrolling viewport that follows the selected row and shows counts above/below instead of letting long model catalogs run off-screen;
  - a failed turn (provider error, turn limit) reports the error and keeps the REPL alive. Without a terminal or with `--json`, an empty prompt is an argument error.
- `--resume-compact <session.jsonl>`: like `--resume` but runs the same summarization compaction before the first turn when the serialized history exceeds 4000 chars (skips with a note otherwise). Works in both REPL and single-shot modes.
- Headless `exec --json` now also emits `tool.call` events alongside `text.delta`/`tool.result`/`completed`. JSON event keys are serialized in alphabetical order.
- Workspace tools: `list_dir`, `read_file`, `search` (ripgrep), `write_file`, `run_command`, `git_status`, and `git_diff`.
  - `list_dir` marks directories with a trailing `/` (like `ls -p`) and the schema says so; `read_file` on a directory explains it is a directory and points to `list_dir`. Models were routinely calling `read_file` on extensionless directories (`src/backend/tests`) and wasting a turn on an unhelpful error.
  - `run_command` enforces a timeout: the model may set `timeout_seconds` (clamped to 1ŌĆō3600 in `agent.rs`), defaulting to `tool::DEFAULT_COMMAND_TIMEOUT_SECS` (120). The child is spawned in its own process group (unix `process_group(0)`; Windows uses `taskkill /T`) so a timeout kills the whole command tree ŌĆö a foreground `node server.js` previously hung a swarm forever (observed live: worker starts the server it just wrote, `run_command` blocks indefinitely). Output is drained on capped reader threads (memory-bounded at the 1 MiB limit instead of buffering unboundedly), and EOF joins use a 2 s `recv_timeout`, which also fixes a pre-existing hang where `Command::output()` blocked on a pipe held open by a background child (`sleep 30 & echo done`). On timeout the error includes the output captured so far, so the model can see what the command printed before it was killed.
  - Every normal and swarm agent is told the exact startup workspace path and warned not to guess container paths or runtimes. The sanitized subprocess PATH includes Apple Silicon Homebrew so a Homebrew-installed `rg` remains usable. `git_status`, `git_diff`, `/diff`, and the swarm final review now treat a non-Git directory as a normal, explanatory result rather than surfacing Git's fatal error; this is separate from shadow checkpoints, which have always supported non-Git workspaces.
- The `--resume` picker: `--resume`/`--resume-compact` with no path (and a TTY) lists this workspace's resumable conversations newest-first with age + first-prompt preview (`session::list_sessions`) and reads a numbered choice. Preview text falls back to the first structured user message for older/message-only sessions. Standalone `swarm_goal` files remain on disk as audit logs but are excluded because they concatenate multiple fresh boss/worker/checker histories and are not valid provider-resumable conversations. An explicit path still resumes directly; a path is only consumed when the next arg is an existing file.
- Provider defaulting: `--provider` is optional. When omitted, `resolve_provider_kind` uses the provider recorded in the newest session (`session::last_provider`; each run records a `provider` event), else the sole available provider, else prints setup guidance and exits. `provider::available_providers()` / `ProviderKind::has_credential()` check cloud keys in env + `.env` + `~/.junebug/credentials.env`, or Ollama runtime reachability.
- `junebug set --provider NAME KEY` saves a credential to `~/.junebug/credentials.env` (0600) then launches the REPL on that provider.
- Safety controls:
  - default permission is now `read-only`;
  - `ask` prompts for writes and commands;
  - `workspace-write` permits only workspace writes; commands always prompt;
  - `yolo` (a.k.a. `approve-all`) grants unrestricted filesystem access (absolute/outside-workspace paths, `.env*`, `.git`, `.junebug`, traversal, and symlinks), lets commands inherit the launching environment, and pre-approves every write and command with no prompt; plan mode still forces read-only over it. Search/Git tools accept optional target paths so work in another repository does not depend on shell `cd` persistence. Set via `--permission yolo` or the `/permissions` menu, which warns in red that protected files/environment secrets may reach the provider and session log. Checkpoints remain scoped to the startup workspace, so outside changes cannot be rewound;
  - outside `yolo`, protected paths include `.env*`, `.git`, and `.junebug` at any depth (component-based, so Windows separators and nested entries are covered);
  - outside `yolo`, path traversal is rejected and symlinks are fully resolved: a symlink inside the workspace (including a dangling one) cannot read or create content outside it;
  - destructive/network-oriented shell commands are classified by a token-based matcher (whole command words plus shell separators, so `git add .` is not flagged while `rm<TAB>-rf` and `echo hi;rm -rf /` are; redirects to `/dev/null` and fd dups like `2>&1` are allowed, file redirects are flagged). **Flagged commands are no longer hard-blocked**: outside `yolo`, classification adds a ŌÜĀ warning to the mandatory approval prompt; yolo intentionally bypasses both warning and approval;
  - all tool dispatch, including MCP tools, is routed through a dedicated `PolicyEngine` (see below) before any side effect runs; the MCP approval prompt shows the exact (truncated) arguments.
- `/keys` + no-model startup: interactive runs with zero usable credentials no longer error ŌĆö the REPL opens in no-model mode (banner `no provider ┬Ę no model ŌĆö /keys to add one`; `/model`, `/compact`, and prompt turns are guarded) and `/keys` (main.rs `handle_keys_command` + masked `read_secret`) picks a provider from a menu, saves the key via `store_credential` to `~/.junebug/credentials.env` (0600), and brings a model up in-place (sticky model else provider default). Keys never enter session logs, the conversation, or the model; the session records only `credential_saved: <provider>`. Non-interactive runs keep the exit-with-guidance behavior. Live-verified over a pty with an isolated HOME (scripts/keys_smoke.exp), including a no-leak grep of the session logs.
- Model swarms (`src/swarm.rs` + `run_swarm`/`handle_swarm_setup` in `main.rs`): the boss/worker/checker orchestration pattern. `/swarm-setup` saves roleŌåÆprovider/model assignments to `~/.junebug/swarm.json` (workspace `.junebug/swarm.json` overrides). `/swarm <goal>` runs sequentially inside the REPL handler (no raw mode; Esc-interrupt does not work mid-swarm ŌĆö Ctrl-C kills the process; checkpoints protect the workspace): (1) boss plans with read-only tools under a hard plan-mode `PolicyEngine`, producing a constitution + fenced-JSON task array (`swarm::parse_tasks`, Ōēż12 tasks, one reformat retry); (2) per task, worker executes (full tools, session permission) then checker verifies (tools minus `write_file`) and must end with a `VERDICT: PASS|FAIL:` line ŌĆö silence counts as failure; failed checks feed back to the worker for up to 2 reworks, then the boss rules (`RULING: WORKER|CHECKER`, default = checker, i.e. unclear rulings never accept unverified work) with one final guided rework; (3) boss reviews outcomes + git diff. Each agent turn is a fresh `run_loop` with a `PinnedModel` and its own message history; the swarm writes its own session file so the main conversation stays resumable, and a summary user/assistant pair is appended to the REPL context afterwards. Not available headless (REPL only, v1). Two resilience layers (added after a live abort on reqwest's "request or response body error"): `swarm_agent` retries a turn up to 2 extra times with 2s/5s backoff when `swarm::is_transient_provider_error` matches (stream death, connection issues, timeouts, 429/5xx ŌĆö each turn rebuilds its fresh message list so retry-from-scratch is safe, and retries are recorded as `swarm_retry` session events); and progress persists to workspace `.junebug/swarm_state.json` (`swarm::SwarmState`: goal, constitution, tasks, per-task outcomes, rework/failure counts) saved after planning and after every task, cleared on completion. `/swarm resume` continues from the first unfinished task (finished ones are listed as "already finished ŌĆö resumed" and their outcome lines are prefilled for the final review); starting a fresh `/swarm <goal>` over saved progress prints a notice that it is being replaced. Abort messages now end with a `/swarm resume` hint.
- Swarm role assignment now uses the same grouped all-configured-provider model picker for boss, worker, and checker; there is no separate provider-selection step.
- File-change diffs (`src/diff.rs`, dependency-free LCS line diff with 2-line context hunks and `┬Ę┬Ę┬Ę` separators): `write_file` changes are shown as colored `-`/`+` hunks in the REPL (via `TurnObserver::on_file_diff` ŌåÆ `TurnEvent::FileDiff`), embedded in `ask`-mode approval prompts (capped at 40 lines), and emitted as `file.diff` JSONL events. Diffs are UI-only ŌĆö never appended to the model context. The pre-write content is captured in `run_loop` before dispatch; denied/failed writes emit no diff. Oversized diffs fall back to block replacement (LCS cell cap) and displays clip at 80 lines.
- Workspace checkpoints (`src/checkpoint.rs`): a shadow Git repo per workspace under `~/.junebug/checkpoints/<hash>` (the user's repo is untouched; non-Git workspaces work). Snapshots are taken before every prompt and, via a `checkpoint` closure threaded through `agent::run_loop`/`execute_tool`, before every permitted mutating tool (after policy/approval, before dispatch ŌĆö denied tools never checkpoint, and this covers `run_command`, which Claude Code's checkpoints famously miss). Unchanged states are deduped (no new tag). `/rewind` lists checkpoints in an arrow-key menu and restores files via `reset --hard` after first checkpointing the pre-restore state, so a rewind is always undoable; conversation history is never rewound. Snapshots respect the workspace `.gitignore` plus a fixed exclude list (`.env*`, `.junebug/`, `.git/`, `target/`, `node_modules/`) ŌĆö proven by tests that secrets are neither captured nor overwritten on restore. Git identity/hooks are pinned per invocation (`-c user.name=ŌĆ” -c core.hooksPath=<absent>`), so nothing depends on or leaks into the user's Git config. Best-effort by design: failures are recorded as `checkpoint_error` session events and never block a tool; `--no-checkpoints` (or missing git) disables the feature with a notice.
- `AGENTS.md` discovery, `.junebug/hooks.json` hooks behind `--enable-hooks`, and local stdio MCP support behind `--enable-mcp`.
- Session continuation with `--resume <session.jsonl>`, and context character-budget compaction with `--max-context-chars`.
- Plan mode has a hard read-only permission guard and filters out write/command tools.
- CI exists at [.github/workflows/ci.yml](.github/workflows/ci.yml) for formatting, tests, and Clippy across macOS, Linux, and Windows.

## Verified evidence

The most recent local gate passed after the `PolicyEngine`/agent-loop extraction:

```sh
cargo fmt
cargo test
cargo clippy --all-targets -- -D warnings
git diff --check
```

As of v0.3.0 there were 38 Rust unit tests plus 3 integration tests. `cargo llvm-cov --summary-only` reported **38.85% line coverage overall** (3022 lines) ŌĆö but that number is dominated by `main.rs` at 0% (955 lines: CLI/REPL glue, exercised only by live `expect` smoke tests, not unit tests) and `editor.rs` at 20% (raw-mode TUI needing a TTY). Excluding `main.rs`, library line coverage is ~57%. The deterministic core is well covered: `policy.rs` 100%, `context.rs` 100%, `markdown.rs` 98%, `session.rs` 81%, `tool.rs` 79%, `agent.rs` 78%. Weakest are the I/O-bound modules that need a live provider/TTY (`provider.rs` 24%, `editor.rs` 20%, `mcp.rs` 43%). To improve the honest number, the next lever is extracting testable logic out of `main.rs` (arg parsing, provider/permission resolution) into the library.

At an earlier point there were 35 Rust unit tests plus 3 integration tests in `tests/policy_integration.rs`. Live DeepSeek smoke tests (driven with `expect` over a pty ŌĆö note `script(1)` is unreliable for this) verified in disposable temp workspaces: streamed Markdown rendering with tool activity lines and approvals; Esc mid-stream interrupting the turn with the follow-up message continuing from the partial reply; `/model` listing live models; `/compact` (11 ŌåÆ 3 messages); `--resume-compact` on a 28 KB session (follow-up turn cost 1371 input tokens instead of ~7k) and its small-history skip path; slash-command and `@file` completion menus including attachment expansion the model could answer from; `list_dir(.)`; and an approved `git log --oneline -5 2>/dev/null || echo none`. Do not use or print credentials from `.env`; it is ignored.

A full-architecture review after the `PolicyEngine` extraction found and fixed these defects (each has a regression test):

1. **Symlink read/write escape.** `checked_path` canonicalized only the parent, so a symlink file inside the workspace could read content outside it, and a dangling symlink could create a file outside it via `fs::write` following the link. The target itself is now fully resolved when it exists (via `symlink_metadata` + `canonicalize`). A follow-up fix: for `list_dir(".")` the parent-ancestor fallback normalized past the workspace root and rejected it; existing targets now short-circuit on their own canonical containment and the ancestor walk applies only to not-yet-existing targets.
2. **Compaction could orphan `tool` messages.** Cutting between an assistant `tool_calls` message and its `tool` results produced requests OpenAI-compatible APIs reject with 400. `context::compact` now widens past the budget until the boundary is valid.
3. **30-second stream death.** `reqwest::blocking::Client::new()` defaults to a 30-second whole-request timeout, killing any streamed turn longer than that. The provider now builds a client with a 30 s connect timeout and 10 min per-turn cap.
4. **Dangerous-command matcher was substring-based.** It blocked `git add .` (contains `dd `) and `ls sudoku/` (contains `sudo`) while missing `rm<TAB>-rf` and `;rm` chains. Replaced with token-based matching.
5. **Invalid JSONL output.** Hand-rolled `json_escape` missed control characters (`\t`, `\r`); exec-mode events are now serialized with `serde_json`. Note the event key order changed (alphabetical); the schema fields are unchanged.
6. **Protected paths were `/`-string-matched.** `.git\config` on Windows and nested `.env`/`.git` entries bypassed protection; matching is now per path component at any depth.
7. **Docs/help drift.** README and `--help` claimed the default permission is `ask`; it is `read-only`. README claimed MCP/hooks are unavailable; both exist behind `--enable-mcp`/`--enable-hooks`. `--plan` was missing from help.
8. Smaller: `write_file` can now create nested parent directories inside the workspace; plan mode no longer connects/offers MCP servers (their calls were all policy-denied anyway).

## Known gaps and risks

These are the highest-priority items, in order:

1. **Complete v0.1 correctness and UX.** The interactive REPL now has streamed Markdown, Esc/Ctrl-C turn interruption, completion menus, `/model`, and `/compact`; still missing are `--cwd`, a diff preview inside `/rewind` (write-time diffs shipped in `src/diff.rs`; restore shipped via checkpoints), precise patch/apply-patch support, command timeouts, output truncation before buffering, tool approval audit records, secret redaction, provider/runtime integration fixtures, long-line wrap handling in the line editor (cursor math assumes one row), and completion of model names after `/model `. Note the editor and turn UI require a real terminal; everything falls back to plain stdio without one.
2. **Documentation drift is fixed as of the architecture review** ŌĆö [README.md](README.md) now documents the real default permission (`read-only`), `--plan`, `--resume`, hooks, and MCP. Keep it in sync. [PLAN.md](PLAN.md) still lists items as deferred because it is the original roadmap, not status tracking.
3. **Refactor into core contracts.** The plan calls for `ModelProvider`, `Tool`, `PolicyEngine`, `SessionStore`, and `PromptBuilder`. `ModelProvider`, `SessionStore` (as `session.rs`), and `PolicyEngine` (as `src/policy.rs`, with dispatch enforcement in `src/agent.rs`) now exist as dedicated modules; `Tool` and `PromptBuilder` are still informal (tool schemas are inline JSON in `main.rs`; there is no dedicated prompt builder beyond string formatting in `main.rs`).
4. **Provider compatibility.** Add and test Anthropic Messages and OpenAI Responses adapters, model capability discovery, retries/backoff, rate-limit handling, custom base URLs, and structured error contracts.
5. **Harden MCP and hooks.** MCP needs server timeout/restart controls, protocol/version tests, tool-level approvals, config provenance/pinning, and remote Streamable HTTP/OAuth/PKCE. Hooks need before/after tool events and allow/deny/observe semantics.
6. **Implement v0.3/v1.** Skills, agent profiles/subagents, worktree isolation, plugin manifests/integrity, local app-server JSON-RPC, policy profiles, packaging/signing, SBOM, docs, and release process are all absent.

## Safe continuation workflow

1. Read [PLAN.md](PLAN.md) completely and treat it as the source of product requirements.
2. Run the verification gate above before and after each meaningful slice.
3. Keep API keys out of patches, output, sessions, and commits. `.env` is intentionally ignored.
4. For every new executable capability, make the default deny/ask decision explicit and add a test proving no unapproved action runs.
5. Prefer a small vertical slice with unit + integration coverage over untested scaffolding.
6. When changing user-facing behavior, update [README.md](README.md) and this document.
